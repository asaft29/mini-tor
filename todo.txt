## Digest / Integrity Check (AES-CTR has no authentication)

Problem: AES-128-CTR provides confidentiality only. A malicious relay can flip bits
in the ciphertext and the receiver will silently decrypt garbage with no way to detect
tampering. Real Tor uses a running SHA-256 digest embedded in every message to detect this.

### Changes required:

1. services/common/src/protocol.rs
   - Add `digest: [u8; 4]` field to Message struct
   - Update to_bytes() to include digest in wire format
   - Update from_bytes() to parse digest field
   - This is a breaking wire format change -- all services must be updated together

2. services/common/src/crypto.rs
   - Add forward and backward digest state to SessionKey (two running SHA-256 hashers)
   - Add update_digest() helper that feeds message bytes into the running hash
   - Add verify_digest() helper that checks received digest against running hash

3. services/tor-client/src/circuit.rs
   - Circuit::send_message() -- update running forward digest, embed 4-byte digest in message
   - spawn_circuit_reader() -- verify backward digest on every received message, drop if mismatch

4. services/relay-node/src/circuit/entry.rs
   - Add per-circuit forward and backward digest state alongside session_key
   - handle_relay() -- verify forward digest on incoming, update backward digest on outgoing
   - handle_backward_relay() -- update backward digest before forwarding to client

5. services/relay-node/src/circuit/middle.rs
   - Same changes as entry.rs

6. services/relay-node/src/circuit/exit.rs
   - Same changes as entry.rs
   - spawn_stream_tasks() -- update backward digest on every data chunk sent back

## Other Missing Features Compared to Real Tor

### 1. Stateful AES-CTR stream (no per-message IV)
Real Tor uses a single stateful AES-CTR stream per circuit direction -- the counter
just keeps incrementing across all messages. This project generates a fresh random IV
for every aes_encrypt() call, adding 16 bytes overhead per encryption layer (48 bytes
per onion-encrypted message due to 3 layers).

Changes required:
- services/common/src/crypto.rs
  - Replace aes_encrypt(data, key) / aes_decrypt(data, key) with a stateful
    AesCtrStream struct that holds the cipher state and counter across calls
  - Remove IV prepending/stripping from encrypt/decrypt
- services/tor-client/src/circuit.rs
  - Circuit holds two AesCtrStream instances per hop (forward + backward) instead
    of deriving a fresh cipher on every call
- services/relay-node/src/circuit/entry.rs, middle.rs, exit.rs
  - Each CircuitContext holds two AesCtrStream instances per circuit

### 2. Fixed-size cells (512 bytes)
Real Tor pads all messages to exactly 512 bytes before sending. Variable-length
messages allow traffic analysis -- an attacker can infer what type of data is being
transferred based on message size patterns.

Changes required:
- services/common/src/protocol.rs
  - Add padding to to_bytes() so every serialized message is exactly 512 bytes
  - Update from_bytes() / from_stream() to strip padding after reading
- All send sites across tor-client and relay-node need no changes (handled in protocol)

### 3. TLS on relay-to-relay connections
Real Tor wraps all relay-to-relay TCP connections in TLS, providing an additional
transport-level encryption layer on top of the onion encryption. This project connects
over plain TCP, so the onion-encrypted cells are visible on the wire (though still
encrypted by the onion layers).

Changes required:
- Add rustls or native-tls dependency to relay-node and tor-client
- services/relay-node/src/main.rs -- wrap TcpListener with TLS acceptor
- services/tor-client/src/circuit.rs -- wrap TcpStream::connect() with TLS connector
- services/common/ -- add shared TLS certificate/key helpers
